### Migration design notes

The orchestrator plan asked for the heuristic seed to live in
`main.dart` `_runUserSettingsMigrations()`. Per the codebase's existing
pattern (versioned `assets/sql/migrations/v$i.sql` files invoked by
`AppDB.migrateDB()` at cold-start), the cleaner path was to bake the
heuristic into `v28.sql` itself. Result is identical (idempotent
`INSERT OR IGNORE` rows on first cold-start) but lives entirely in the
schema migration layer â€” no main.dart edits needed.

The transactions-table recreate dance preserves every column, FK, default,
and CHECK constraint verbatim from the v18 schema (the last recreate of
the transactions table). I cross-referenced both `tables.drift` and every
migration v18â†’v27 to confirm:
- No indexes on `transactions` (verified against `app_db.g.dart` and
  every `*.sql` migration â€” all `CREATE INDEX` statements target other
  tables: `attachments`, `pendingImports`, `statementImportBatches`).
- No triggers on `transactions` anywhere in the codebase.
- No column additions to `transactions` between v18 (the last recreate)
  and v27 except those already in the v18 shape.

Therefore the v28 recreate is safe â€” the `transactions_temp` definition
is byte-identical to the v18 shape modulo the widened CHECK on
`exchangeRateSource`.

### Design decisions captured at implementation time

1. **`rateSourceForPair` shape**: the design's signature was
   `RateSource? rateSourceForPair(String from, String to)` but the policy
   does NOT itself read `preferredRateSource` (it's not in the
   resolver's input set). To avoid coupling the policy to
   `UserSettingService`, the policy accepts an optional named parameter
   `preferredVesRateSource` that the caller supplies (defaults to
   `RateSource.bcv` when null). This matches the design intent â€” the
   resolver knows mode/primary/secondary, but the *current preferred*
   BCV/Paralelo choice belongs to the caller's stream composition. Phase
   6 widget integration will combine the policy stream with the
   `preferredRateSource` stream to feed this argument.
2. **Pure `buildPolicy` exposed**: design said "private builder", but I
   exposed it as `static buildPolicy(...)` (the `Rx`-bound `_buildPolicy`
   wrapper stays private). Rationale: the unit tests in Task 2.4 need to
   exercise the resolution table without a Drift DB, and this pure entry
   point keeps the test surface trivial (no streams, no fakes). The
   private wrapper preserves the design's "factory binds to streams"
   intent for production callers.
3. **`equivalenceCurrency` accessor**: the orchestrator brief asked for
   `String? equivalenceCurrency()` (method). I made it a getter for
   consistency with `showsEquivalence` and `showsRateSourceChip` (also
   getters). Behaviour is identical; switching to a method is trivial
   if Phase 6 prefers it.
4. **No widget wiring**: per the orchestrator constraint, no
   `StreamBuilder<CurrencyDisplayPolicy>` lands in any widget yet. The
   resolver singleton is unused at runtime â€” only by tests. Phase 6 owns
   the actual consumption.

### What was implemented

#### Bug 1 â€” `RateSourceBadge` always shows "Auto" (Phase 6 risk #2)

**Type**: Model issue (NOT a codegen issue, NOT a schema issue).

**Root cause**: The custom Dart `ExchangeRate` class at `lib/core/models/exchange-rate/exchange_rate.dart` extends `ExchangeRateInDB` (which already has the nullable `source` column from the v28 schema). However, the custom constructor accepted only `id, date, currency, exchangeRate` and never forwarded `source` to `super`. Because the `getExchangeRates` / `getLastExchangeRates` Drift queries use `WITH ExchangeRate` (the custom class), the codegen generated `ExchangeRate(id: ..., date: ..., currency: ..., exchangeRate: ...)` â€” passing nothing for `source`. Result: `er.source` was always `null` at runtime, so `RateSourceBadge`'s null-fallback always rendered "Auto".

**Fix**: One-line change to the custom model â€” added `super.source,` to the constructor parameter list. Then re-ran `dart run build_runner build --delete-conflicting-outputs`. The drift_dev codegen automatically picked up the new optional named param and generated `source: row.readNullable<String>('source'),` in BOTH `getExchangeRates` (line 10146) and `getLastExchangeRates` (line 10162) of `app_db.g.dart`. No `.drift` file change needed â€” `SELECT e.*` / `SELECT er.*` already projected the `source` column; only the model wiring was missing.

#### Bug 2 â€” `excRate` reference in `transaction_filter_set.dart` (Phase 9 + Phase 6 risk #1)

**Type**: Cross-query SQL alias coupling.

**Root cause**: `TransactionFilterSet.toTransactionExpression()` returns the same predicate regardless of which query consumes it. The `minValue` / `maxValue` `CustomExpression`s referenced `excRate.exchangeRate` â€” an alias defined in the `latestRates` CTE join of `getTransactionsWithFullData`. Phase 9 reshaped `countTransactions` to drop both the CTE and the `excRate` join (it now returns native sums grouped by currency). When the same predicate carrying `minValue` / `maxValue` flowed into `countTransactions`, the resulting SQL referenced an undefined alias â†’ runtime error.

**Fix chosen â€” Option A (Phase 9's recommended migration)**: Replace `COALESCE(excRate.exchangeRate, t.exchangeRateApplied)` with `t.exchangeRateApplied`. Rationale:
- `t.exchangeRateApplied` is a column on the `transactions` table itself (no JOIN needed) so the predicate works in BOTH `getTransactionsWithFullData` and `countTransactions` without any SQL-side restructuring.
- `t.exchangeRateApplied` is the immutable per-row rate captured at insert time (per `transactions/spec.md` "ADDED Requirement: `exchangeRateApplied` inmutable por transacciÃ³n"). Filtering a transaction by its locked-in converted value is also a more correct semantic than filtering by today's possibly-changed rate â€” the user's intent when applying a min/max value filter is "this is what the transaction was worth at the time", not "this is what the value would be at today's rate".
- Same-currency-as-preferred path (`a.currencyId = preferredCurrency`) still short-circuits to `1.0`, preserving the existing semantic.

This avoids Option B's blast radius (restoring a CTE inside `countTransactions` would re-couple it to the rate-source ladder Phase 9 deliberately decoupled) and avoids the temporary disable suggested in the task brief â€” Option A is the cleanest path per the existing code structure.
